| 类别 | 禁止 |
|------|------|
| 废话开场 | 有一说一、说实话、其实 |

## 爆款内容模式 (2026-02-20)

### 来自@yanhua1010的12.2万浏览推文分析

**数据表现：**
- 22回复 / 233转发 / 812点赞 / 1678收藏 / 12.2万浏览

### 标题公式
```
数字 + 痛点/场景 + 价值承诺
```
示例：100小时OpenClaw使用经验：你的24小时AI员工完整指南

### 内容结构
1. **hook** - 用一句话抓住注意力
2. **是什么** - 一句话说清楚
3. **能做什么** - 列出具体能力
4. **案例** - 有细节的故事
5. **反直觉观点** - 与众不同的见解
6. **实操建议** - 可执行的步骤
7. **总结** - 核心观点回顾

### 反向提示技巧
> "根据你对我的了解，你觉得我们应该做什么？"

比"帮我做X"更高效，因为AI有更多上下文。

### 适用到风水内容
- "今天运势不错，应该发什么内容？"
- "根据最近的风水热点，应该写什么？"

## SkillsBench：写好Skills的7个原则 (2026-02-20)

### 来自@yibie的2万浏览推文分析
研究：84任务 × 7,308轨迹

### 核心发现
| 发现 | 效果 |
|------|------|
| 好的Skills | +51.9%成功率 |
| 差的Skills | -39.3%成功率 |
| 模型自己写 | 负效果 |

### 7个原则

**1. 人工编写 > 模型生成**
- 人工：+16.2% | 模型生成：-1.3%
- 模型生成的步骤太笼统、缺细节

**2. 少即是多（数量）**
- 2-3个Skills最优：+18.6%
- 4+个反而下降：+5.9%

**3. 详细 > 全面（长度）**
- 控制在800-1500 tokens
- Comprehensive（全面）反而-2.9%

**4. 程序性 > 陈述性**
- ❌ "Python是什么"（是什么）
- ✅ "用pandas读取CSV的三步流程"（怎么做）

**5. 领域差异巨大**
- Healthcare：+51.9%（模型缺乏知识）
- Manufacturing：+41.9%
- Software Engineering：+4.5%（模型已有）

**6. 模块化与组合**
- 适用一类任务，不是单一实例
- 可编辑、版本控制、组合使用

**7. 验证与迭代**
- 配套验证方法
- 成功率<70%需重写

### 核心启示
> "差距不在模型能力，而是Skill边界上的经验工程"

### SKILL.md模板
```yaml
---
name: "skill_name"
description: "一句话描述（20字内）"
---

## 核心步骤（3-5步）
每步包含可执行代码和关键参数说明
## 验证清单
- [ ] 检查点1

```

## Skill设计实战经验 (2026-02-18)

### 主人踩坑总结

1. **参数要对齐** - 复杂项目容易对不上，增加校验
2. **文档要仔细** - 瞄一眼就用出问题难查
3. **规则会遗漏** - 关键规则多重强调
4. **规则要放宽** - 慢慢打磨，别太死

### 高确定性场景（Hiroki经验）

- 代码优先：确定性逻辑用代码
- 验证环节：输出后增加复核
- Hybrid：代码+Agent各司其职

## 任务追踪系统 (2026-02-19)

### 架构
参考 VoxYZ 的 closed-loop 架构：
```
提案(proposal) → 审批 → 任务(mission) → 执行 → 事件(event) → 触发
```

### 文件结构
```
tasks/
├── config.json           # 策略配置（每日限额）
├── proposals.json        # 提案
├── missions.json         # 任务
├── events.json          # 事件日志
├── tweet-drafts.json   # 推文草稿
├── task-tracker.js     # 核心脚本
├── tweet-workflow.cjs  # 发推工作流
└── track               # 命令行封装
```

### 使用命令
```bash
./track proposal tweet '内容'   # 创建提案
./track tweet draft '内容'      # 创建推文草稿
./track tweet quota            # 查看配额
./track stats                 # 任务统计
./track stale                 # 检查卡住任务
```

### 配额配置
- 推文：3条/天
- 小红书：2条/天
- 超额自动排队

## 单Agent vs 多Agent (2026-02-20) - 来自@karry_viber

### 核心观点
- 5只不如1只深度进化
- 单Agent深度进化 > 多Agent分工

### 5份记忆 vs 1份记忆
- 多Agent：5个角色各有独立MEMORY.md，隔离干净
- 问题：没有任何角色真正认识主人
- 单Agent：完整记忆，知道你的一切，替你做决策

### 隔离悖论
- 隔离越好，信息损耗越大
- 每一次@转发都是信息压缩
- 语气、偏好、言外之意，传着传着就丢了

### 养🦞不是开公司
- 多Agent = 组织架构
- 单Agent = 养搭档
- 不会跟5个人聊所有事，会跟同一个人聊所有事

### 两个阶段
| 阶段 | 模式 | 关注点 | 工具 |
|------|------|--------|------|
| 第一阶段 | 多Agent | 效率 | Claude Code |
| 第二阶段 | Single Agent深度进化 | 理解和信任 | OpenClaw |

### 核心理念
- "一个人 + 一只深度进化的🦞 = 一支团队"
- "不是多，是深"
- OpenClaw护城河：住在手机里，记得你的一切，24小时替你行动

## OpenClaw完全指南 (2026-02-20) - 来自@Jason Zhu 71.6万浏览

### 来源
- @Jason Zhu / @GoSailGlobal
- 71.6万浏览，5041书签
- 基于 AlexFinn 的教程

### 核心概念

**反向提示（Reverse Prompting）**：
- 传统提示：告诉AI做什么
- 反向提示：问AI应该做什么
- "基于你对我们的了解，我们应该做什么？"
- 把反向提示贴在显示器上

**大脑与肌肉架构**：
- Opus做决策（大模型）
- Codex/其他模型执行（便宜）
- 省钱又高效

**让它自我改进**：
- 不纠正它，让它自己构建解决方案
- 任何做不到的事，让它构建出来

### 三个安全规则
1. OpenClaw可以访问电脑一切
2. 不要暴露给外界（防提示注入）
3. 深思每个指令的后果

### 我们可以借鉴
- 加强反向提示使用
- 大脑+肌肉架构（Kimi决策+DeepSeek执行）
- Mission Control概念

## AI写作协作方法 (2026-02-20) - 来自@范凯说AI

### 四步改稿法
1. 换心态 - 别把AI当"全文编辑器"，当成分工明确的合作者
2. 先给编辑意图说明书 - 核心判断 + 目标读者 + 不满意点
3. 按逻辑模块分段提交 - 一段一段来
4. 每轮只做一件事 - 一次只优化一个点

### 七步共创框架
1. 定目标 - 目标读者 + 解决什么问题 + 核心判断
2. 做蓝图 - 标题备选 + 一句话主张 + 核心分论点
3. 扩成段落级大纲 - 每个分论点拆到具体
4. 分块共写 - 一节一节往前走
5. 质量检查 - 一句概括本节观点？有没有案例？
6. 统一精修三轮 - 动结构 → 抛光表达 → 平台适配
7. AI写反驳再回应 - 从反对者视角补强

### 核心观点
- "AI不是提高效率——它在重构人类的创造方式"
- "关键不在AI，而在你自己有没有'灵魂'"

### 应用
- 直接套用到风水推文创作
- 先想核心观点，再分段让AI辅助

## 去AI味终极方案 (2026-02-20) - 来自@宝玉 30.4万浏览

### 核心观点
- "去不掉AI味，只能用自己味道盖过"

### 问题根源
1. 提示词是一次性的，用完就扔
2. 从来没告诉AI要像谁
3. 所有人用同一套提示词，又变同质化

### 解决方案：写作风格Skill（菜谱）

### 四步建立菜谱
1. 让AI尝菜 - 把3-5篇满意文章发给它分析
2. 让AI做菜 - 按Skill写，你来尝

1. 角色与读者 - 基调

- 废话开场白：有一说一、说实话、其实

## 记忆系统优化 (2026-02-20) - 来自@OneHopeA9

### 核心改进：先结构再工具
qmd很强，但先要有好的记忆结构。正确顺序：
1. 先做结构化（L0/L1/L2 + TTL）
2. 再上 qmd 做检索引擎

### .abstract 机制（L0优化）
- 目录里有哪些主题
- 每个主题对应哪些文件
- 最近更新时间
- 检索关键词（tags）

### 记忆检索协议
```
1. 先读 memory/.abstract
2. 判断主题范围
3. 调 qmd query 获取 top-k 片段
4. 必要时再打开对应 L1/L2 原文
5. 回答后把新增关键信息写回 L2
```

### 自动化cron配置
1. 每小时或每2小时：qmd embed增量更新
2. 每天凌晨：memory-janitor（处理P1/P2过期）
3. 每周一次：L2 → L1 compact + 刷新 .abstract

### 常见坑Tips
1. 只上qmd，不做分层 → 前期爽，后面失控
2. .abstract太长 → L0变L2，失去索引意义
3. 不做TTL → P1/P2变相永久，系统再次膨胀
4. 不回写记忆 → 只会读不会写，系统不会进化

### 实测收益
- 日常对话token：通常降5~10x
- 回忆相关性：明显提升
- 长期稳定性：不再越跑越胖
